## Review — Red-team Round 2: cache_sim.py / Experiment 5

### Correct (verified)
- **Queue recompute** (`cache_sim.py:48-51`) is correct: `diskq` times recomputed from `current[1]` on every queue mutation; `_advance` pops with consistent precomputed times.
- **Oracle tl fix** (line 63-67): oracle uses `true[tok0+b, tl]` with `tl=l+1`, eid built with `tl*E` — no longer prefetching the current layer. Real-trace semantics verified against `model.py:162` (`true = topks[l + h]`) and `dump_traces.py:37-42`.
- **Demand-priority / starvation**: prefetches are only ever issued into an empty queue+idle disk (line 65), so prefetches can never queue ahead of demand; demand fetches re-front correctly (lines 92-95). No starvation path exists by construction.
- **Published synthetic numbers reproduce exactly** (0.877/0.880/0.882/0.888 tok/s; waste 68.3/63.8/53.5) — the code does what RESULTS.md says it does. The problem is that what it does is wrong.

### Blocker
1. **Synthetic predictor is misaligned by one layer — the entire synthetic table is invalid.** `synth_traces` (`cache_sim.py:120-134`) sets `pred[t,l]` = noisy copy of `true[t,l]` (same layer), but `Sim.prefetch` (line 63-68) treats `pred[t,l]` as a prediction *for layer l+h* and builds `eid=(l+h)*E+pe`. Measured realized next-layer precision: **15.2% / 15.9% / 17.0%** for the claimed base 0.83 / joint 0.89 / oracle 1.0. Even the "oracle" is a 17% predictor; its lower "waste" (53.5) comes only from Zipf popularity coincidence. All synthetic numbers (RESULTS.md:196-204, findings 2-4) are artifacts. Running the one-line fix (`pred[t,l] = noisy(true[t,l+1])`) flips the magnitudes by an order: **demand 0.876 → base 1.053 (+20%), joint 1.086 (+24%), oracle 1.161 (+33%) tok/s; waste 34.4/23.7/0.0 MB/tok** (oracle waste → 0, as it must be). The conclusion "effect sizes +0.5-3% tok/s" does not survive the fix; the thesis ordering and "~25-40% of base-to-oracle headroom" (25% buggy → 31% fixed) do. Fix: one line in `synth_traces`; rerun.

### High
2. **Toy headline "learned prefetch LOSES" is a strawman-engine artifact.** Fetch time 0.676 ms > B=1 compute window 0.1 ms, so every prefetch overruns into the next layer and demand queues behind it (the `prefetch()` comment at line 66-67 claims it "never risks delaying demand fetches" — false; it only avoids *starting* while busy). A fits-in-window guard (`FETCH_MS <= T_LAYER*B`) makes learned prefetch exactly match demand at B=1 (85.8/86.1 tok/s — it simply never prefetches). More importantly, **B=8 flips the sign**: learned h1 *wins* at C=64 (92.9 vs 88.4 demand, +5%), ties at C=128, oracle +20%. `results_cache_sim.csv` contains only B=1 rows despite `--batches` defaulting to `1,8`, and RESULTS.md never states the batch size for the toy table. Fix: fits-in-window guard + report B=8.
3. **The monotonicity claim is contradicted by their own CSV.** RESULTS.md:210-212 claims "demand < base acc < joint acc < oracle, monotonically, at every budget and horizon." In `results_cache_sim.csv:3-4` (C=64): joint h1 = 62.49 < base 62.71; h2 joint 66.47 < base 66.49; and demand (85.8) beats *all* learned policies at every toy budget/horizon. The ordering only holds in the (bugged) synthetic table.

### Medium
4. **Waste-reduction claim is fragile twice over.** The published "7-10% cut" (68.3→63.8) is a layer-shift artifact. Under the fixed predictor with uniform errors the cut is 31% (34.4→23.7); with popularity-correlated (Zipf) mispredictions — more realistic than independent uniform — absolute waste drops ~3x (34.4→12.3 base) while the ratio holds (~34%). So the *ratio* is robust to the error model, but every absolute MB/tok figure and the published percentage are not. Uniform-iid errors at E=256 are worst-case; real mispredictions skew popular and are sometimes useful later.
5. **Missing dimensions, ranked by threat:** (a) **TTFT/prefill** — RESULTS.md itself (finding 5) admits the headline benefit may live there, unmeasured; (b) **B=1-only synthetic** — at serving batch B≥32 with E=256, top-8, the per-layer union approaches E and cache economics flip (C=2000 = 10% of 19200 slots becomes hopeless; pinning dominates); (c) hot-store pinning interaction; (d) QD>1/io_uring — least threatening for 19MB transfer-bound reads (QD doesn't add bandwidth once saturated), but the serial QD=1 model with 0.1 ms latency is the pessimistic end of "honest" for the 2.88MB toy geometry (real TLC random-read latency ~10-80µs; 0.1ms fits QLC).

### Low
6. **Waste accounting understates absolutely, symmetric across policies:** end-of-run flush (`cache_sim.py:113-115`) counts only in-flight `diskq` entries; unused prefetches still cache-resident at end escape. The cumulative `used` set (line 44, 60) also exempts any re-fetch of a once-used expert forever.
7. **Hit-rate accounting inconsistent at B>1:** `reqs` incremented per request (line 78) but in-flight-prefetch hits counted once per unique eid (line 95) → hit_rate understated at B>1 (contributes to the odd hit=0.000 at B=8/C=64).
8. **No seeds/CIs:** synthetic is seed=0, ntok=2000; toy is a single 32k trace. With the bug fixed the tok/s effects are large enough to survive noise, but waste ratios are seed-sensitive.

### Answer to "does 'prediction barely helps' survive a better engine model?"
No — but the primary reason is internal, not the engine: the synthetic scenario measured a 15-17% predictor, not an 83-100% one. With the intended predictor, even this conservative one-fetch-per-window, QD=1 engine shows +20-33% tok/s in the Colibri geometry. The one-per-window policy itself is *not* binding in the synthetic regime (0% of windows skipped due to busy-at-start; 3.9ms fetch ≈ 4ms window), so Colibri's async pool would not have changed the synthetic verdict — the toy geometry is where the engine policy (fits-in-window, B>1) reverses the sign.
